When both align it places a live market order on BitGet with automatic stop-loss and take-profit levels.

🚀 Going Live — Step by Step

1 Fork the repo

Fork ballikov-lgtm/BotTrading on GitHub, then clone it locally and run npm install.

2 Create your BitGet API key

In BitGet → Profile → API Management → Create API:

  • Permissions: Read-Write Futures Orders + Read-Write Spot Trading
  • Passphrase: alphanumeric only — no $, @, ! or special characters. They break environment variable parsing.
  • IP whitelist: leave blank unless you have a static IP
  • Do NOT enable withdrawal or copy-trading permissions
⚠️ "Futures Position" read permission is not available on most account tiers. The bot works around this by using environment variables for position tracking — you don't need it.

3 Set BitGet to one-way position mode

BitGet defaults to hedge mode. The bot requires one-way mode.

Go to: BitGet → Futures → Settings (⚙️) → Position Mode → select One-Way.

⚠️ Skipping this step causes error 40774 on every order — no trades will fire.

4 Set leverage on BitGet manually

The bot tries to set leverage via API but the set-leverage endpoint requires a separate permission that isn't always available. Manually set 3× isolated on each pair you plan to trade. You only need to do this once per symbol.

5 Create the logs branch on GitHub

Railway pushes bot state to a logs branch after each run so you can see activity without triggering a redeploy. Create it before deploying:

git checkout -b logs
git push origin logs
git checkout main

6 Create a GitHub Personal Access Token

GitHub → Settings → Developer Settings → Personal Access Tokens → Fine-grained:

  • Repository access: your trading repo only
  • Permissions: Contents → Read and Write

Save the token — you'll add it to Railway as GITHUB_TOKEN.

7 Deploy to Railway

Create a new project on Railway, connect your GitHub repo, then add these environment variables:

# BitGet credentials
BITGET_API_KEY=your_key
BITGET_SECRET_KEY=your_secret
BITGET_PASSPHRASE=alphanumeric_only

# Must be the string "false" — not blank, not 0
IRONCLAD_PAPER=false

# Position sizing
PORTFOLIO_USD=500
MAX_TRADE_USD=50
LEVERAGE=3
MAX_TRADES_PER_DAY=3

# GitHub state sync
GITHUB_TOKEN=your_github_pat
⚠️ Critical: IRONCLAD_PAPER must be set to the string false. If the variable is missing or empty the bot defaults to paper mode and no real orders are placed — even on Railway.
✅ Railway auto-deploys when you push to main. The runner starts immediately and loops every 15 minutes.

8 Verify it's working

After the first Railway run, check the logs branch on GitHub — ironclad-log.json should be updated with scan results. If it's there, the bot is running and syncing correctly.

🔧 Common Errors & Fixes

Error CodeMessageFix
40404 Endpoint not found API endpoint must use kebab-case: place-order not placeOrder. Already fixed in the bot code.
400172 Margin mode cannot be empty Add marginMode: 'isolated' to the order body. Already fixed in the bot code.
400172 Side parameter mismatch Hedge-mode side format (open_long) doesn't work in one-way mode. Use side: 'buy'/'sell' + tradeSide: 'open'. Already fixed.
40774 Unilateral position type BitGet account is in hedge mode. Switch to one-way mode in BitGet Futures settings.
40014 Incorrect permissions (set-leverage) Non-fatal — the bot catches this and continues. Set leverage manually on BitGet instead.
Paper mode Bot never places real orders Set IRONCLAD_PAPER=false (exact string) in Railway environment variables.
