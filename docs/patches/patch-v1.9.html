✦ Latest — Patch v1.9

TP Limit Orders — Fixed

Take-profit orders now correctly sit on the Bitget orderbook as pending limit orders at your target prices. Three bugs were causing all TP orders to silently fail or fire immediately at market. This patch is cumulative — one page takes you from any previous version to v1.9.

Released 9 May 2026  ·  Works from any previous version (v1.0–v1.8)

What Changed

All changes from v1.5 through v1.9 are included. If you're on any earlier version, this single patch page covers everything.

Three stacked bugs were causing all TP orders to fail since v1.7

Positions opened but no take-profit limit orders appeared on Bitget. The root causes have now been identified and fixed. No account changes are required — this is a code-only update.
New in v1.9
🚨
Critical Fix

Missing posSide field — Bitget hedge-mode positions require it on close orders

Bitget's hedge-mode uses posSide (not holdSide or nothing) to identify which position a close order targets. Without it, every limit close attempt returned [22002] No position to close even though the position clearly existed. Adding posSide: 'long' or posSide: 'short' to the order body fixes this completely.

🚨
Critical Fix

Wrong endpoint — /close-positions is a market close, not a limit order

An attempted workaround switched to the /close-positions endpoint, which appeared to work but was actually executing an immediate market close of the position — not placing a pending limit order on the book. This caused positions to close at entry price within seconds of opening. The bot is back on /place-order with the correct posSide field.

🚨
Critical Fix

Parallel TP placement — Bitget only allows one pending close order per position

All three TP orders were fired simultaneously with Promise.all. Bitget accepts the first and rejects the other two with [22002] because a pending close order is already in flight. Fixed: orders are now placed sequentially with a 2-second gap between each one.

🔧
Fix

TP levels validated against actual fill price before placement

Previously TPs were calculated from the signal price and passed straight to the order placer. If the actual market fill came in higher than the signal price (common with market orders), TP1 could end up below the actual entry — an impossible target. The bot now fetches the confirmed fill price from Bitget before placing TP orders and recalculates any invalid level using R:R fallbacks (1.5R / 2.5R / 4.0R).

Included from v1.8
📍
Improvement

Entry prices record Bitget's actual fill price

The bot fetches the confirmed fill price from Bitget after a market order lands and records that instead of the signal price.

🔗
Improvement

Position monitoring upgraded — Holdings permission + safe fallback

Uses the Holdings API permission to detect SL/TP closures. Falls back gracefully if permission is missing.

📈
Improvement

Compound position sizing

Position size scales with your portfolio balance each run rather than the fixed starting value.

Included from v1.7
🎯
Improvement

Real TP limit close orders + fill monitoring + auto-trail SL

TP levels fire as real limit close orders on Bitget. SL trails to break-even after TP1, and to TP1 price after TP2.

Included from v1.6
🚨
Critical Fix

Ghost order prevention — failed orders no longer block the daily limit

Failed API orders were being logged to the CSV, counting against the daily trade limit. Fixed: orders are only logged when a real orderId is returned.

⚡
Improvement

Leverage set inline — no separate API call needed

Leverage is passed directly in the place-order body, removing a dependency on a rarely-granted API permission.

Included from v1.5
🚨
Critical Fix

Bitget API bugs — orders now actually fire

Three silent API failures blocked all live orders on v1.0–v1.4. Watchlist expanded from 10 to 18 pairs.

Code Reference

What the fixed placeLimitClose function looks like in v1.9.

The critical change is adding posSide to the order body. This tells Bitget's hedge-mode engine which position side the close order belongs to:

async function placeLimitClose(symbol, side, qty, price) {
  // posSide = direction of the EXISTING position (long or short)
  // side    = opposite (sell to close long, buy to close short)
  // Without posSide, hedge-mode returns [22002] "No position to close"
  const body = {
    symbol, productType: 'USDT-FUTURES', marginCoin: 'USDT',
    marginMode: 'isolated',
    side:      side === 'long' ? 'sell' : 'buy',
    posSide:   side,   // ← the fix
    tradeSide: 'close',
    orderType: 'limit',
    price:     price.toString(),
    size:      qty.toString(),
  };
  const r = await bitgetRequest('POST', '/api/v2/mix/order/place-order', body);
  if (r.code !== '00000') {
    console.log(`⚠️  Limit close FAILED @ $${price}: ${r.code} ${r.msg}`);
    return null;
  }
  return r.data?.orderId || null;
}

TP orders are placed sequentially (2 seconds apart) because Bitget only allows one pending close order per position at a time. TPs are also validated and recalculated against the actual fill price before being placed:

// Fetch actual fill price from Bitget before placing TPs
const fillDetail = await getOrderDetail(symbol, orderId);
const fillPrice = parseFloat(fillDetail?.priceAvg) || entry;

// Validate each TP — replace any on the wrong side of fill price
const validFor = p => isLong ? p > fillPrice * 1.001 : p < fillPrice * 0.999;
const fixedTp1 = validFor(tp1) ? tp1 : fallback(1.5);  // 1.5R
const fixedTp2 = validFor(tp2) ? tp2 : fallback(2.5);  // 2.5R
const fixedTp3 = validFor(tp3) ? tp3 : fallback(4.0);  // 4.0R

// Sequential placement — 2s between each to avoid [22002] on 2nd and 3rd
const tp1Id = await placeLimitClose(symbol, side, qty40, fixedTp1);
await new Promise(r => setTimeout(r, 2000));
const tp2Id = await placeLimitClose(symbol, side, qty35, fixedTp2);
await new Promise(r => setTimeout(r, 2000));
const tp3Id = await placeLimitClose(symbol, side, qty25, fixedTp3);
