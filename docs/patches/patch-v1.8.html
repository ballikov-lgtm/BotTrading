✦ Latest — Patch v1.8

Accurate Fill Prices + Position Monitoring + Holdings Fix

Entry prices now record Bitget's actual fill price. The bot can now properly detect SL/TP closures using the Holdings API permission. Compound position sizing added for growing portfolios. This patch is cumulative — one page takes you from any previous version to v1.8.

Released 9 May 2026  ·  Works from any previous version (v1.0–v1.7)

⚠️ Required Account Change First

This must be done before deploying the code or position monitoring will not work.

Bitget API key update required — enable "Holdings" permission

v1.8 uses the Bitget position-read endpoint to detect when your SL or TP fires. This requires a permission Bitget labels "Holdings" (not "Read" or "Trade" — those are different things). Without it the bot cannot tell when a position has closed and it will stay "open" in the tracker indefinitely.

How to enable it:
1. Log in to Bitget → click your profile icon (top right) → API Management
2. Find your existing API key → click Edit
3. Under USDT-M Futures: tick the Holdings toggle
4. Save. No need to regenerate the key — the change takes effect immediately.

What Changed

All changes from v1.5 through v1.8 are included. If you're on any earlier version, this single patch page covers everything.

New in v1.8
📍
Improvement

Entry prices now record Bitget's actual fill price

Previously the bot logged the signal price (the price at the moment the signal fired) as the entry price. In practice Bitget fills market orders at a slightly different price. The bot now fetches the confirmed fill price from Bitget immediately after an order lands and records that instead. Your CSV and dashboard will now always show what you actually paid.

🔗
Improvement

Position monitoring upgraded — Holdings permission + safe fallback

The bot now correctly calls Bitget's all-positions endpoint using the Holdings permission. If the permission is missing (code 40014) it returns a clear sentinel value rather than silently failing, and falls back to checking pending orders — so live positions are never incorrectly closed due to a permissions gap. The audit script also now flags 40014 as a hard failure (not a warning) so it's obvious before a trading session.

📈
Improvement

Compound position sizing — grows with your portfolio

Position size is now calculated from the current portfolio value each run rather than the fixed PORTFOLIO_USD starting value. As the portfolio grows from winning trades, each new position automatically scales up proportionally. Risk per trade remains fixed at 1%.

Included from v1.7
🎯
Improvement

Real TP limit close orders + fill monitoring + auto-trail SL

Take-profit levels fire as real limit close orders on Bitget. The bot monitors fill status each cycle. Stop loss trails to break-even after TP1 and to TP1 price after TP2.

Included from v1.6
🚨
Critical Fix

Ghost order prevention — failed orders no longer block the daily limit

Failed API orders were being logged to the CSV, counting against the daily trade limit. Fixed: orders are only logged when a real orderId is returned.

⚡
Improvement

Leverage set inline in order body — no separate API call needed

The separate set-leverage API call requires an extra permission that standard futures keys don't have. Leverage is now passed directly in the place-order body.

🔧
Fix

Margin mode on close orders — prevents errors 400172 and 22002

marginMode: 'isolated' is now always specified on close orders to match how positions are opened.

Included from v1.5
🚨
Critical Fix

Bitget API bugs fixed — orders now actually fire

Three silent API failures blocked all live orders on v1.0–v1.4: wrong endpoint casing, wrong side format for one-way mode, wrong margin mode. Watchlist expanded from 10 to 18 pairs.

How to Apply This Patch

Three steps — estimated time 10 minutes including the Bitget account change.

Enable Holdings permission on your Bitget API key

See the red box at the top of this page for the exact steps. This is required before the code update will work correctly.

⚠️ Do this first. If you deploy the code without enabling Holdings, the bot will log a clear warning each cycle and fall back to safe mode (never false-closing a live position) — but SL/TP detection will be limited until the permission is enabled.

Pull the latest code from GitHub

In your local repo folder:

git pull origin main

This brings in the updated bot-ironclad.js (fill price recording, Holdings-aware position monitoring, compound sizing) and audit.js (40014 now flags as hard failure).

✅ All v1.8 changes are committed to the repo. git pull is all you need for the code.

Redeploy to Railway

Trigger a redeploy so Railway picks up the new code:

git commit --allow-empty -m "Deploy patch v1.8" && git push origin main

Or in the Railway dashboard: your service → Redeploy. After the first cycle, check Railway logs for ✅ all-position: X positions found — that confirms Holdings is working.

Code Reference

Key changes in bot-ironclad.js for transparency.

Actual fill price recording after order placement

// Before (v1.7) — signal price used as entry:
appendTrade({ entry: signalPrice, ... });

// After (v1.8) — fetch confirmed fill price from Bitget:
const detail = await bitgetRequest('GET',
  `/api/v2/mix/order/detail?symbol=${symbol}&orderId=${orderId}`, null);
const fillPrice = parseFloat(detail?.data?.priceAvg) || signalPrice;
appendTrade({ entry: fillPrice, ... });  // real fill price, not signal

Position monitoring — Holdings permission + safe sentinel

// After (v1.8) — clear sentinel on 40014, never returns ambiguous null:
async function fetchAllBitgetPositions() {
  const r = await bitgetRequest('GET',
    '/api/v2/mix/position/all-position?productType=USDT-FUTURES&marginCoin=USDT', null);
  if (r.code === '00000') return r.data || [];
  if (r.code === '40014') {
    console.log('  ⚠️  Holdings permission missing — enable in Bitget API Management');
    return 'NO_READ_PERMISSION';  // sentinel: caller knows to fall back safely
  }
  return null;  // network / other error
}

// checkPositions() handles the sentinel — never false-closes a live position:
if (bitgetAll === 'NO_READ_PERMISSION') {
  stillOpen.push(...livePositions);  // keep all open — safe default
}

Compound position sizing

// Before (v1.7) — fixed starting portfolio value:
const risk = CONFIG.portfolioUsd * 0.01;

// After (v1.8) — current portfolio value, grows with wins:
const currentPortfolio = await getCurrentPortfolioValue(); // sum of balance + unrealized
const risk = currentPortfolio * 0.01;  // 1% of actual current value
